Introduction

This project is a real-time multiplayer puzzle game where multiple players compete on a shared 4×4 grid. Players take turns placing colored shapes on the board, trying to match 3 or more blocks in a row to score points. The game uses Node.js, Express.js, Socket.IO, and EJS to create an interactive web experience. A unique feature is the dual-window design: the main page acts as a lobby where players enter their names, and each player gets their own game window to play in.

The main goal was to build a web application that handles multiple players at once, keeps everyone's game view synchronized, and provides a smooth user experience. The system needed to handle various situations like players disconnecting, preventing duplicate usernames, dealing with slow internet connections, and managing browser popup blockers. The game also includes visual feedback when players score points, making the experience more engaging and fun.

System Architecture and Technology Stack

The game uses a client-server setup. The server runs on Node.js with Express.js handling web pages, and Socket.IO managing real-time communication between players. The server keeps track of everything: the game board, who's playing, whose turn it is, and everyone's scores. All game rules are checked on the server to prevent cheating and make sure everyone sees the same game state. The server uses a special tracking system to link each player's connection to their identity, preventing someone from joining twice with the same connection.

The website has three pages built with EJS templates: a home page, an about page, and the game page. The game page is smart - it changes based on how you access it. If you go to /game directly, you see the lobby with a name input box. If you open it with a name in the URL like /game?name=PlayerName, it becomes a game window that automatically connects you to the game. The game board uses SVG graphics to draw shapes, and the page updates instantly when anything happens. The site also has a theme switcher that lets you choose between light mode, dark mode, or follow your system settings.

Core Features and Implementation

The lobby system is the heart of the multiplayer experience. When you visit the main game page, you enter your name and click join. The page then opens a new window just for you with your name in the URL. This gives each player their own clean game interface without the lobby clutter. Multiple people can join from the same lobby page one after another. If your browser blocks popups, the game shows a helpful message telling you how to fix it.

Player management has several safety checks to keep the game fair. Before you can join, the system checks if your name is already taken - this happens even before opening your game window, so you know right away if you need to pick a different name. The server also makes sure you can't accidentally join twice from the same connection. Once you're in, the server tracks your player ID, adds you to the player list, sets your score to zero, and tells everyone else you've joined. If you disconnect or quit, the server removes you from the game, adjusts whose turn it is if needed, and updates everyone else. There's also a 60-second timer - if you don't make a move during your turn, you're automatically removed so the game can continue.

The game itself is turn-based. Each turn, the server randomly picks a block (with a shape and color) and gives it to the current player. You can only place blocks in empty spaces, and only during your turn. If you try to click an occupied space, you get an instant error message without affecting other players. After you place a block, the server checks if you made any matches - 3 or more blocks in a row with the same color or shape. Matched blocks disappear and you earn points. If the board fills up completely, it clears and you get a 16-point jackpot bonus. The game includes detailed instructions explaining the rules, shapes, and colors to help new players understand quickly.

Everything updates in real-time using Socket.IO messages. When the board changes, everyone sees it instantly. When it's your turn, only you see which block you need to place - other players just see whose turn it is. The scoreboard updates immediately when someone scores. When players join or leave, everyone gets notified. The game also has visual feedback: when you match blocks, they light up with an animation, a big score popup appears in the center of the screen, and your name flashes in the scoreboard. Jackpots get even more dramatic effects with gold colors and bigger animations.

Technical Challenges and Solutions

The dual-window system had some tricky parts. The main challenge was making the same page work as both a lobby and a game window. The solution was to check the URL - if it has a player name in it, the page knows it's a game window and hides the navigation menu automatically. Browser popup blockers were another issue since many browsers block new windows by default. The game handles this by showing a clear message telling users to allow popups. The system also prevents you from joining multiple times from the same connection by tracking which connections already have players.

Preventing rapid clicking was important. At first, if you clicked really fast, the game would send multiple requests before the server could respond. This was fixed by locking clicks after you place a block - you can't click again until the server confirms your move and starts the next turn. If the server takes too long (more than 3 seconds), you get a timeout message and can try again.

Keeping everyone's game synchronized was complex. The solution was to make the server the boss - all game changes happen on the server first, then get sent to all players. The server checks everything: is it your turn? Is that space empty? Is your move valid? This prevents cheating and makes sure everyone sees the same game, even if someone has a slow internet connection. Error messages only go to the player who made the mistake, not to everyone.

Handling disconnections during someone's turn needed careful planning. When a player disconnects, the server immediately removes them, updates the turn order, and moves to the next player. Everyone gets notified about who left. The system works the same whether you quit on purpose or your connection drops unexpectedly.

A critical timing issue was fixed where new game windows would try to join before the connection was ready. Originally, the join request was sent as soon as the page loaded, but the Socket.IO connection might not be established yet. This caused the duplicate username warning to fail. The fix was to wait for the connection to fully establish before sending the join request, ensuring all error messages work properly.

Testing and Validation

The game went through lots of testing. Basic gameplay was tested: joining, leaving, taking turns, placing blocks, scoring, and winning. The dual-window system needed special testing - making sure the lobby opens game windows correctly, checking that game windows auto-connect, and verifying that menus hide properly. Multiple browser windows were used to simulate several players at once, testing things like fast clicking, multiple people joining at the same time, duplicate names, trying to join twice, and people leaving mid-game. Edge cases were also tested: clicking occupied spaces, clicking during someone else's turn, filling the whole board, and dealing with popup blockers.

Connection testing checked how the game handles different internet conditions. Tests included slow connections, temporary disconnections, and complete connection loss to make sure the game stays stable. The synchronization was verified by comparing what different players see after a series of moves - everyone should see the exact same board and scores. Special attention was given to the connection tracking system to make sure it properly links players to their connections and cleans up correctly when people leave.

User experience testing looked at how responsive and clear the interface is. Testers confirmed the lobby-to-game-window flow felt natural and that having separate windows helped focus on the game. The design was tested on different screen sizes from phones to desktop monitors to make sure everything works and looks good everywhere. The theme system was checked to ensure it detects your system preferences correctly and remembers your choice. The game also supports keyboard navigation and has good color contrast in both light and dark modes.

Learning Outcomes and Reflection

This project taught me a lot about real-time web technologies, especially Socket.IO. I learned the difference between regular web requests (where you ask and get an answer) and persistent connections (where the server can send you updates anytime). The event-based programming style required thinking carefully about timing, error handling, and keeping everyone's game state in sync. Building the dual-window system taught me about managing browser windows, using URL parameters, and keeping different parts of the app separate but connected.

Managing game state across multiple players was more complex than expected. Handling edge cases like slow internet, duplicate names, and disconnections required careful planning. Making the server the authority for all game rules was the right choice - it prevents cheating and keeps everyone synchronized. I learned that preventing duplicate entries and validating everything on the server is crucial for multiplayer games.

Working with JavaScript's asynchronous nature and Node.js helped me understand how to handle multiple connections at once. Managing many players, each with their own state and events, showed me both the power and challenges of async programming. The project reinforced the importance of good error handling, especially when dealing with user input, network issues, and browser limitations like popup blockers. I also learned to think about which information should go to everyone versus just specific players, which improves both security and user experience.

Conclusion

This project successfully demonstrates a real-time multiplayer web game using modern web technologies. The system handles multiple players at once, keeps everyone's game synchronized, and provides an engaging experience with responsive design and instant updates. The dual-window design - separating the lobby from game windows - is an innovative approach that helps players focus on the game. The challenges faced during development, like preventing rapid clicking issues, keeping everyone synchronized, checking for duplicate names, and handling browser compatibility, provided valuable learning experiences and made the final product more reliable.

The game meets all requirements of the JC2503 Web Application Development course, showing practical skills in server programming with Node.js and Express, real-time communication with Socket.IO, client-side JavaScript for page interactions, EJS templates, and responsive CSS design. The project demonstrates both technical skills and problem-solving abilities in handling multiplayer game challenges like connection tracking, duplicate prevention, and window management. Future improvements could include saving game data to a database, user accounts and login, multiple game rooms, spectator mode, game replays, and different game modes or board sizes to make the game even more engaging.
