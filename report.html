Introduction

This project implements a real-time multiplayer puzzle game that allows multiple players to compete simultaneously in a shared 4×4 grid environment. The application demonstrates the practical implementation of modern web technologies including Node.js, Express.js, Socket.IO, and EJS templating. The game features turn-based mechanics where players place colored shapes on a grid, attempting to create matching patterns to earn points. The system handles real-time synchronization across multiple clients, manages player sessions, and provides an interactive user interface with responsive design. A distinctive architectural feature is the dual-window system where the main game page serves as a lobby for player registration, while individual game windows are spawned for each participant, creating isolated yet synchronized gaming experiences.

The primary objective of this project was to develop a full-stack web application that showcases real-time communication capabilities while implementing complex game logic and state management. The application needed to handle multiple concurrent users, ensure consistent game state across all clients, and provide an intuitive user experience. Additionally, the system required robust error handling for edge cases such as player disconnections, rapid clicking, network latency issues, and duplicate player registration attempts. The window management system needed to gracefully handle browser popup blockers and maintain proper state isolation between the lobby interface and individual game windows.

System Architecture and Technology Stack

The application is built on a client-server architecture using Node.js as the runtime environment and Express.js as the web application framework. The server handles HTTP requests for static pages and establishes WebSocket connections through Socket.IO for real-time bidirectional communication. The backend maintains the authoritative game state, including the grid configuration, player registry, current turn information, and scoring data. All game logic executes server-side to prevent cheating and ensure consistency across clients. A critical architectural component is the socket-to-player mapping system implemented using a Map data structure, which enforces a one-to-one relationship between WebSocket connections and player identities, preventing duplicate registrations from the same connection.

On the frontend, the application uses EJS as the templating engine to render three main pages: a home page with basic introduction, an about page with personal information, and the game page containing the interactive puzzle interface. The game page implements a dual-mode interface that adapts based on URL parameters. When accessed directly at /game, it displays as a lobby with full navigation, player name input, and a join button. When accessed with a name parameter such as /game?name=PlayerName, the interface transforms into a dedicated game window where navigation links are hidden, and the Socket.IO connection automatically initiates player registration. This parameter-driven conditional rendering is achieved through client-side JavaScript that inspects window.location.search and dynamically adjusts DOM element visibility. The client-side JavaScript handles user interactions, renders the game grid dynamically using SVG graphics for shapes, and listens for Socket.IO events to update the interface in real-time. The styling is implemented with custom CSS featuring a comprehensive theme system that supports light mode, dark mode, and system preference detection, with smooth transitions and responsive layouts that adapt to different screen sizes.

Core Features and Implementation

The lobby and window management system represents a key architectural innovation in the application. The main game page at /game functions as a centralized lobby where users enter their desired player names and click a join button. Upon submission, the client-side JavaScript invokes window.open() with a constructed URL containing the player name as a query parameter, spawning a new browser window or tab dedicated to that player's game session. This approach provides several advantages: each player receives an isolated interface focused solely on gameplay without lobby clutter, multiple players can join sequentially from the same lobby page, and the separation of concerns between registration and gameplay improves code maintainability. The system includes user-friendly error handling for popup blockers, displaying an alert message instructing users to enable popups if the browser prevents window creation.

The player management system forms the foundation of the multiplayer functionality and implements multiple layers of validation to ensure game integrity. When a user joins the game, the server first checks if the connecting socket already has an associated player ID in the socketPlayerMap, rejecting duplicate join attempts from the same connection with an appropriate error message. Next, the server validates that the requested player name is not already in use by scanning the active players array, preventing name conflicts that could confuse participants. Only after passing these checks does the server assign a unique player ID, establish the socket-to-player mapping, add the player to the active roster, initialize their score entry, and broadcast the updated player list to all clients. When players disconnect, either intentionally through the quit button or due to network issues, the server automatically removes them from the active player list, clears the socket mapping, adjusts the turn order if necessary, and broadcasts updates to all remaining clients. A timeout mechanism removes inactive players after 60 seconds of inactivity during their turn, ensuring the game continues smoothly even when participants become unresponsive.

The game logic implements a turn-based system where players take sequential turns placing blocks on the 4×4 grid. Each block has two attributes: a shape (circle, square, star, or triangle) and a color (red, blue, yellow, or green). The server maintains a pool of all 16 possible block combinations and randomly assigns one to the active player at the start of each turn. Players can only place blocks in empty grid cells, and only during their designated turn. The server validates placement attempts by checking both turn ownership and cell availability, emitting a placementError event to the requesting client if they attempt to place a block in an occupied cell, providing immediate feedback without disrupting other players' experiences. After a successful placement, the system checks for scoring conditions by examining rows, columns, and diagonals for three or more consecutive blocks matching either color or shape. Matched blocks are removed from the grid and returned to the pool, earning points for the player. If the grid becomes completely filled, it resets and awards a jackpot bonus of 16 points.

Real-time synchronization is achieved through Socket.IO's event-driven architecture. The server emits events such as gridUpdate to broadcast board changes, newTurn to signal turn transitions, scoreUpdate to refresh the leaderboard, playLists to update the player roster, and joinError or placementError to communicate validation failures to specific clients. Importantly, the current block information is only sent to the active player to prevent other participants from seeing upcoming moves, adding a strategic element to the game. The client-side code listens for these events and updates the DOM accordingly, re-rendering the grid, updating the scoreboard, and displaying turn indicators. Error events trigger user-friendly alert dialogs that explain the issue without requiring page reloads or disrupting the game flow.

Technical Challenges and Solutions

Implementing the dual-window architecture presented unique challenges in maintaining proper state isolation while ensuring synchronized gameplay. The primary difficulty was distinguishing between lobby access and game window access using the same route endpoint. This was solved by implementing URL parameter inspection in the client-side JavaScript, where the presence of a name query parameter triggers automatic player registration and conditional hiding of navigation elements. Browser popup blockers posed another challenge, as many modern browsers block window.open() calls by default for security reasons. The solution involved providing clear user feedback through an alert message when popup creation fails, instructing users to enable popups for the site. Additionally, the system needed to prevent users from accidentally joining multiple times from the same window, which was addressed through the socketPlayerMap validation that rejects duplicate join attempts from already-registered socket connections.

One significant challenge encountered during development was preventing race conditions when players clicked rapidly on grid cells. Initially, multiple click events could be registered before the server responded, causing the client to send duplicate placement requests. This was resolved by implementing a client-side locking mechanism that disables further clicks immediately after a placement attempt. The lock is released when the server broadcasts the next turn event, or after a timeout period if the server becomes unresponsive. A timeout alert notifies users if the server doesn't respond within three seconds, allowing them to retry the operation without permanently freezing the interface.

State synchronization presented another complex challenge, particularly ensuring all clients maintained identical views of the game board despite varying network conditions. The solution adopted a server-authoritative model where the server acts as the single source of truth. All game state modifications occur server-side, with clients receiving updates through broadcast events. The server validates every action, checking turn ownership, grid cell availability, and move legality before applying changes. This approach prevents client-side manipulation and ensures consistency even when clients experience packet loss or temporary disconnections. The validation layer includes checks for duplicate player names and duplicate socket registrations, with specific error messages communicated back to the requesting client through targeted socket.emit() calls rather than broadcasts, ensuring error feedback reaches only the affected user.

Handling player disconnections during their turn required careful consideration of game flow continuity. The system detects disconnections through Socket.IO's disconnect event and immediately removes the player from the active roster, clearing the corresponding entry in the socketPlayerMap to free the socket for potential future connections. If the disconnected player was currently taking their turn, the server automatically advances to the next player and initiates a new turn. The turn index is adjusted to account for the removed player's position in the rotation, ensuring the turn order remains correct for remaining participants. All clients receive notifications about the disconnection and updated player lists, maintaining transparency about the current game state. The system distinguishes between intentional quits via the quit button and unexpected disconnections, providing appropriate feedback messages in each case.

Testing and Validation

The application underwent extensive testing across multiple dimensions. Functional testing verified core gameplay mechanics including player joining and leaving, turn rotation, block placement validation, scoring calculations, and win condition detection. The dual-window architecture required specific testing scenarios: verifying that the lobby page correctly spawned game windows with proper URL parameters, confirming that game windows automatically initiated player registration upon loading, and ensuring navigation elements were appropriately hidden in game windows while remaining visible in the lobby. Multiple browser instances and windows were used to simulate concurrent players, testing scenarios such as rapid successive moves, simultaneous join attempts from different windows, attempts to join with duplicate names, attempts to join multiple times from the same socket connection, and mid-game disconnections. Edge cases like attempting to place blocks in occupied cells, clicking during other players' turns, filling the entire grid, and handling popup blocker interference were specifically tested to ensure proper handling and user feedback.

Integration testing focused on the Socket.IO communication layer, verifying that events were properly emitted and received under various network conditions. Tests included simulating slow connections, temporary network interruptions, and complete disconnections to ensure the system remained stable and recovered gracefully. The state synchronization mechanism was validated by comparing game states across multiple clients after sequences of moves, confirming that all participants maintained identical views of the board and scores. Special attention was given to testing the socket-to-player mapping system, ensuring that the one-to-one relationship was properly maintained and that cleanup occurred correctly during disconnections and quits.

User experience testing evaluated the interface responsiveness, visual clarity, and overall usability. The dual-window system was assessed for intuitiveness, with testers confirming that the lobby-to-game-window flow felt natural and that the separation improved focus during gameplay. The responsive design was tested on different screen sizes from mobile devices to desktop monitors, ensuring the grid layout and controls remained functional and aesthetically pleasing across all viewports. The theme system was verified to correctly detect system preferences and persist user selections across page reloads, with the theme toggle button functioning consistently across both lobby and game window contexts. Accessibility considerations were addressed by implementing keyboard navigation support and ensuring sufficient color contrast in both light and dark modes.

Learning Outcomes and Reflection

This project provided valuable hands-on experience with real-time web technologies, particularly the WebSocket protocol and Socket.IO library. Understanding the differences between traditional HTTP request-response patterns and persistent bidirectional connections was crucial for implementing the multiplayer functionality. The event-driven programming model required careful consideration of event ordering, error handling, and state consistency across distributed clients. The implementation of the dual-window architecture offered insights into browser window management APIs, URL parameter handling, and the challenges of maintaining state across multiple browser contexts while ensuring proper isolation between lobby and gameplay interfaces.

State management in a distributed system proved to be more complex than initially anticipated. Designing a robust synchronization mechanism that handles edge cases like network latency, packet loss, duplicate registrations, and client disconnections required deep thinking about system architecture and failure modes. The server-authoritative approach, while adding complexity to the backend logic, ultimately provided the most reliable solution for maintaining game integrity. Implementing validation layers for both player names and socket connections taught valuable lessons about data integrity and the importance of preventing duplicate entries in multiplayer systems.

Working with asynchronous JavaScript and Node.js event loops enhanced understanding of non-blocking I/O operations and callback patterns. Managing multiple concurrent connections, each with its own state and event handlers, demonstrated the power and challenges of asynchronous programming. The project also reinforced the importance of proper error handling and defensive programming, particularly when dealing with user input, network communication, and browser API limitations such as popup blockers. The experience of implementing targeted error messaging versus broadcast updates highlighted the importance of considering which clients need which information, improving both security and user experience.

Conclusion

This project successfully demonstrates the implementation of a real-time multiplayer web application using modern web technologies. The system effectively handles multiple concurrent users, maintains consistent game state across clients, and provides an engaging user experience through responsive design and smooth real-time updates. The dual-window architecture separating lobby functionality from gameplay interfaces represents an innovative approach to multiplayer game design, improving user focus and interface clarity. The challenges encountered during development, particularly around race condition prevention, state synchronization, duplicate registration prevention, and browser compatibility issues, provided valuable learning opportunities and resulted in a more robust final implementation.

The application meets all core requirements of the JC2503 Web Application Development course, showcasing practical skills in server-side programming with Node.js and Express, real-time communication with Socket.IO, client-side JavaScript for DOM manipulation and event handling, template rendering with EJS, and responsive CSS design. The project demonstrates not only technical implementation capabilities but also problem-solving skills in addressing complex distributed systems challenges such as maintaining one-to-one socket-player mappings, implementing validation layers for duplicate prevention, and managing window-based user interfaces. Future enhancements could include persistent data storage with databases, user authentication systems, multiple concurrent game rooms with room management, spectator modes, replay functionality, and additional game modes or grid sizes to further expand the application's functionality and user engagement.
